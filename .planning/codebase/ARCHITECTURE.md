# Architecture

**Analysis Date:** 2026-04-16

## Pattern Overview

**Overall:** Monolithic MVC with Server-Side Rendering via Inertia.js

**Key Characteristics:**
- Laravel backend with RESTful resource routing
- Vue 3 frontend rendered server-side via Inertia v3
- NativePHP Mobile for iOS/Android deployment
- Type-safe frontend with Wayfinder typed route generation
- Model-observer pattern for automatic data transformations
- Eloquent ORM with scoped queries for default sort ordering

## Layers

**Presentation (Frontend):**
- Purpose: Mobile-first UI for vehicle data management
- Location: `resources/js/pages/`, `resources/js/components/`
- Contains: Vue 3 pages, Vuetify 4 Material Design components, composables
- Depends on: Inertia adapter, Wayfinder route helpers, i18n
- Used by: End users via NativePHP mobile app

**Request Handler (Controllers):**
- Purpose: HTTP request routing, validation, response serialization
- Location: `app/Http/Controllers/`
- Contains: Seven resource controllers (`VehicleController`, `VehicleRefuelController`, `VehicleServiceController`, `VehicleServiceTypeController`, `VehicleServiceReminderController`), plus `ReportController` and `SettingsController`
- Depends on: Request validation, Eloquent models, Resource transformers
- Used by: Inertia frontend via HTTP

**Data Transformation (Resources):**
- Purpose: Consistent JSON serialization from models to frontend
- Location: `app/Http/Resources/`
- Contains: `VehicleResource`, `VehicleRefuelResource`, `VehicleServiceResource`, `VehicleServiceTypeResource`, `VehicleServiceReminderResource`
- Pattern: Eloquent resources that shape model data for Inertia props

**Business Logic (Models & Services):**
- Purpose: Domain logic, relationships, and report generation
- Location: `app/Models/`, `app/Services/`
- Contains:
  - `Vehicle` (central aggregate with UUID PK)
  - `VehicleRefuel` (fuel tracking with odometer)
  - `VehicleService` (maintenance records linked to reminders)
  - `VehicleServiceReminder` (maintenance schedules by odometer intervals)
  - `VehicleServiceType` (maintenance category/icon definitions)
  - `Report::fuelCosts()` (static method for chart data generation)
- Depends on: Database (Eloquent query builder)
- Used by: Controllers, data transformation

**Middleware/Infrastructure:**
- Purpose: Cross-cutting concerns (i18n, appearance, Inertia setup)
- Location: `app/Http/Middleware/`
- Contains:
  - `HandleInertiaRequests` - shares locale, primaryColor, colorScheme, flash messages
  - `HandleLocale` - reads locale cookie and sets app locale
  - `HandleAppearance` - manages theme and color preference persistence

**Observers & Scopes:**
- Purpose: Model lifecycle hooks and default query ordering
- Location: `app/Observers/`, `app/Models/Scopes/`
- Contains:
  - `VehicleObserver` - auto-uppercases plate_number on save
  - Five default sort scopes (one per model) applied via `#[ScopedBy(...)]` attribute

**Frontend Routing (Type-Safe):**
- Purpose: Type-safe route generation eliminating hardcoded URLs
- Location: `resources/js/routes/`
- Contains: Auto-generated Wayfinder route functions (index, create, store, show, edit, update, destroy) for each resource
- Pattern: Each route exports functions with `.url()`, `.form()`, `.post()`, `.put()`, `.delete()` methods

**Frontend Actions (Controller Bridge):**
- Purpose: Typed representation of backend actions for IDE autocomplete
- Location: `resources/js/actions/`
- Contains: Mirrored action structure from PHP controllers (generated by Wayfinder)

## Data Flow

**Vehicle Creation (User Perspective):**

1. User navigates to `/vehicles/create` (rendered server-side)
2. `VehicleController::create()` responds with `Inertia::render('vehicles/Create')`
3. Frontend renders `vehicles/Create.vue` → renders `vehicles/Form.vue` (reusable form)
4. User fills form and submits via `Form.vue::submit()` → calls `form.post(store.url())`
5. HTTP POST to `/vehicles` reaches `VehicleController::store()`
6. Request validated, model created via `Vehicle::create($validated)`
7. `VehicleObserver::saving()` fires, auto-uppercases `plate_number`
8. Database commit, redirect to `/vehicles` with success flash
9. Frontend receives redirect and navigates back to vehicle list

**Vehicle Service Reminder Trigger (Odometer-Based):**

1. User logs service for vehicle (creates `VehicleService` record)
2. Service record includes `odometer` reading and `vehicle_service_type_id`
3. Frontend displays reminders via `vehicle-service-reminders/Index.vue`
4. Each reminder computed:
   - `latestVehicleServiceOdometer()` - queries latest service of that type
   - `recommendedVehicleServiceOdometer()` = latest + `every` interval
   - `overdueOdometerDiff()` = recommended - vehicle.currentOdometer()
   - `isOverDue()` = true when currentOdometer >= recommended
5. Resource transforms these computed values for display

**Report Generation (Fuel Costs Over Time):**

1. User navigates to `/reports` with optional date range and vehicle filter
2. Query params validated in `ReportController::show()`
3. Calls `Report::fuelCosts($from, $to, $vehicleId)`
4. Query chains:
   - Groups `VehicleRefuel` by date and vehicle_id
   - Filters by date range
   - Selects MAX(unit_price) per date per vehicle
5. Transforms into Chart.js structure:
   - `labels` = unique dates
   - `datasets` = per-vehicle with random color via `colority()` helper
6. Returns to frontend as Inertia prop in `Reports.vue`

**Settings Persistence (Cookies):**

1. User changes locale/color/theme in `Settings.vue`
2. Form submits to `/settings/locale`, `/settings/color`, or `/settings/theme`
3. `SettingsController` stores in HTTP-only cookie
4. `HandleInertiaRequests` middleware reads cookies and shares to frontend
5. `MobileLayout.vue` watches props and applies via `useAppTheme()` composable

**State Management:**

- Server state (vehicles, refuels, services): Fetched on each page render via Inertia props
- Client state (settings, theme, locale): Shared via Inertia props from cookies
- Form state: Managed by Inertia's `useForm()` hook (reactive, handles loading/errors)
- URL state: Not explicitly managed; route parameters drive single-resource views

## Key Abstractions

**Vehicle (Aggregate Root):**
- Purpose: Represents a single vehicle with all related data
- Examples: `app/Models/Vehicle.php`
- Pattern: UUID primary key, computed `full_name` attribute, `currentOdometer()` method aggregates latest odometer from refuels and services

**VehicleServiceReminder (Business Domain):**
- Purpose: Encodes maintenance scheduling logic (interval-based by odometer)
- Examples: `app/Models/VehicleServiceReminder.php`
- Pattern: Methods like `isOverDue()`, `overdueOdometerDiff()`, `recommendedVehicleServiceOdometer()` encapsulate the business rules

**Resource Transformers:**
- Purpose: Decouple model structure from frontend contract
- Examples: `app/Http/Resources/VehicleResource.php`
- Pattern: Each resource calls `toArray()` to shape model data, includes computed properties (e.g., `current_odometer`, `full_name`)

**Wayfinder Route Helpers:**
- Purpose: Type-safe, autocompleted frontend route generation
- Examples: `resources/js/routes/vehicles/index.ts`
- Pattern: Auto-generated via Wayfinder, every route function has `.url()` and `.form()` methods, eliminates hardcoded string URLs

## Entry Points

**Backend:**
- Location: `routes/web.php`
- Triggers: HTTP requests from frontend
- Responsibilities: RESTful resource routing, redirect to `/vehicles` index, settings endpoints

**Frontend:**
- Location: `resources/js/app.ts`
- Triggers: Page load or navigation
- Responsibilities: Initialize Inertia, register plugins (i18n, Vuetify), mount Vue app to DOM

**Mobile App:**
- Location: `nativephp/` (iOS, Android native wrappers)
- Triggers: App launch on device
- Responsibilities: Load Laravel server as webview, handle native OS integration

## Error Handling

**Strategy:** Request validation at controller boundary, no silent failures

**Patterns:**
- Controllers use `Request::validate()` with rule arrays
- Validation errors returned as HTTP 422 with error messages
- Frontend displays validation errors via `form.errors` object
- Flash messages for success/error feedback (persisted in session)
- No explicit exception handling visible; Laravel's exception handler returns appropriate responses

## Cross-Cutting Concerns

**Logging:** Implicit via Laravel's default logging (not configured in CLAUDE.md; check `config/logging.php`)

**Validation:** 
- Backend: `Request::validate()` with rule sets
- Frontend: Vuetify form components display errors from `form.errors`
- Pattern: Fail fast with clear validation feedback

**Authentication:** 
- Not visible in routes/controllers examined; likely handled by `auth` middleware (not shown in web.php)
- Assumes authenticated user context for all operations

**Localization:** 
- Via cookies stored by `SettingsController`
- `HandleLocale` middleware reads cookie and applies via `App::setLocale()`
- Frontend uses Vue i18n with `resources/js/i18n/index.ts` (It/En locales)

**Appearance/Theme:**
- Vuetify theme applied via `useAppTheme()` composable in `MobileLayout.vue`
- Primary color and color scheme stored in cookies
- Shared to frontend via Inertia middleware props
